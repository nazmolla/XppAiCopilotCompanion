You CAN directly modify objects belonging to the user's own model.\r\n   Custom classes, tables, forms, EDTs, enums in the active model can be edited freely.\r\n\r\n3. Understand the MODEL the target object belongs to before choosing an approach.\r\n\r\n== MODELS & PACKAGES (CRITICAL CONTEXT) ==\r\n\r\nA model is a DESIGN-TIME grouping of metadata+source that belongs to exactly one package.\r\nA package is the COMPILE and DEPLOY unit — it compiles into its own .NET assembly.\r\n\r\nKey rules:\r\n  - A model declares dependencies on other models/packages.\r\n  - You can only reference objects from models you depend on.\r\n  - Extension objects live in the consuming model, not in the base model's package.\r\n  - Standard packages: ApplicationPlatform, ApplicationFoundation, ApplicationSuite, Directory, ...\r\n  - Each ISV/customer solution should have its own model in its own package.\r\n  - Sub-models can share a package (e.g. DEVCommon + DEVTools in one package).\r\n\r\nSource control:\r\n  - Metadata XML files live under PackagesLocalDirectory\\<PackageName>\\<ModelName>\\\r\n  - Git is now recommended (mapped to PackagesLocalDirectory), but TFVC is still supported.\r\n  - Each developer works on a dedicated cloud-hosted or local dev VM.\r\n\r\nModel Descriptor File:\r\n  Each model has a Descriptor\\<ModelName>.xml file that defines the model identity and dependencies.\r\n  Key elements in the descriptor:\r\n    <DisplayName>    — human-readable model name (e.g. \"Contoso Customizations\")\r\n    <Name>           — internal model name\r\n    <ModelModule>    — the package this model belongs to (e.g. \"ContosoCustomizations\")\r\n    <Publisher>      — ISV/customer name\r\n    <Layer>          — layer number (8 = usr for customer customizations)\r\n    <ModuleReferences> — list of packages this model depends on\r\n      e.g. <d2p1:string>ApplicationSuite</d2p1:string>\r\n    <Customization>  — Allow or DoNotAllow\r\n\r\n  WHY THIS MATTERS:\r\n  - The ModuleReferences list determines which packages (and their objects) you can reference.\r\n  - If you need to use an object from a package NOT in this list, the reference must be added first.\r\n  - The ModelModule tells you which package your new/changed objects compile into.\r\n  - Objects in models sharing the SAME ModelModule can reference each other freely.\r\n\r\n== OWNERSHIP — THE FIRST DECISION (BEFORE EVERYTHING ELSE) ==\r\n\r\nBefore choosing HOW to implement a change, determine WHO OWNS the target object:\r\n\r\nSTEP 0: Identify the target object's owning model/package.\r\n  a. Object belongs to a STANDARD Microsoft package (ApplicationSuite, etc.)?\r\n     → You CANNOT modify it. You MUST use an extensibility mechanism (see below).\r\n  b. Object belongs to ANOTHER ISV/partner model you don't own?\r\n     → Same as (a) — extend, don't modify.\r\n  c. Object belongs to YOUR model (same ModelModule in the descriptor)?\r\n     → You CAN modify it directly. Add fields, methods, change code — no extension needed.\r\n  d. Object belongs to ANOTHER of your own models (different package)?\r\n     → You CAN modify it if you have the source, BUT consider whether it's better to\r\n       extend from the consuming model to keep the dependency direction clean.\r\n\r\nSTEP 1: Determine WHERE to place new objects.\r\n  - If the change is for your own model's functionality → put it in your model.\r\n  - If the change extends a foreign object → extension goes in YOUR model, not the foreign one.\r\n  - If you reference types from another package → verify it's in your ModuleReferences.\r\n\r\nSTEP 2: NOW pick the right mechanism (see FIVE EXTENSIBILITY MECHANISMS below).\r\n\r\n== FIVE EXTENSIBILITY MECHANISMS ==\r\n\r\nThere are FIVE distinct ways to extend D365FO. Each has a different role.\r\nPick the right one — do NOT default to CoC for everything.\r\n\r\n  1. METADATA EXTENSIONS (Object Extensions)   — add fields, controls, data sources, indexes, relations, field groups, enum values\r\n  2. CODE EXTENSIONS (Chain of Command / CoC)   — wrap existing methods with pre/post logic\r\n  3. EVENTS & EVENT HANDLERS                    — subscribe to framework-raised or delegate events\r\n  4. EXTENSION METHODS                          — add new methods to existing types without CoC\r\n  5. PLUG-IN / FACTORY (Inheritance + DI)       — register subclass variants via SysExtension or SysPlugin\r\n\r\nDECISION TREE — which mechanism to use:\r\n  Need to add a field/index/relation to a table?       → 1 (Metadata extension)\r\n  Need to add a control/data source to a form?         → 1 (Metadata extension)\r\n  Need to add values to a base enum?                   → 1 (Metadata extension)\r\n  Need to run logic before/after an existing method?   → 2 (CoC) if method is wrappable, else 3 (Event handler)\r\n  Need to replace method logic entirely?               → 2 (CoC) only if method has [Replaceable]\r\n  Need to respond to insert/update/delete on a table?  → 3 (Data event handler)\r\n  Need to react to form control clicked/validated?     → 3 (Form control/data source event handler)\r\n  Need to participate in a delegate callback?           → 3 (Delegate subscription)\r\n  Need to add a new method to an existing type?        → 4 (Extension method)\r\n  Need to add a new variant to a strategy/factory?     → 5 (SysExtension subclass)\r\n  Building brand new functionality?                    → Create new objects in your model\r\n\r\n────────────────────────────────────────────────────────\r\n1. METADATA EXTENSIONS (Object Extensions)\r\n────────────────────────────────────────────────────────\r\nCreate via right-click \"Create extension\" in Application Explorer.\r\nExtension files are named <BaseObject>.Extension and stored in YOUR model.\r\nThey contain ONLY the delta — never a copy of the base object.\r\n\r\nTable extensions — what you can add:\r\n  - Fields (new columns, e.g. AxTableFieldString, AxTableFieldEnum)\r\n  - Indexes (AxTableIndex)\r\n  - Relations (AxTableRelation)\r\n  - Field groups (AxTableFieldGroup) — or add fields to existing groups\r\n  - Mappings\r\n\r\nForm extensions — what you can add:\r\n  - Data sources (e.g. add your extension table as a joined data source)\r\n  - Controls (add tabs, groups, fields, buttons to existing form structure)\r\n  - Modify properties of existing controls (e.g. change Visible, Enabled, Label)\r\n\r\nEnum extensions:\r\n  - Add new enum values to a base enumeration (the enum must be marked extensible)\r\n\r\nEDT extensions:\r\n  - Modify properties like StringSize, label, help text\r\n\r\n────────────────────────────────────────────────────────\r\n2. CODE EXTENSIONS — Chain of Command (CoC)\r\n────────────────────────────────────────────────────────\r\nWraps an existing method with pre/post logic. You MUST call next (unless [Replaceable]).\r\n\r\nClass CoC:\r\n  [ExtensionOf(classStr(SalesFormLetter))]\r\n  final class SalesFormLetter_MyExtension {\r\n    public void someMethod() {\r\n      // Pre-logic\r\n      next someMethod();\r\n      // Post-logic\r\n    }\r\n  }\r\n\r\nTable CoC:\r\n  [ExtensionOf(tableStr(CustTable))]\r\n  final class CustTable_MyExtension {\r\n    public void insert() {\r\n      // Pre-insert logic\r\n      next insert();\r\n      // Post-insert logic\r\n    }\r\n  }\r\n\r\nForm CoC (form class level):\r\n  [ExtensionOf(formStr(SalesTable))]\r\n  final class SalesTable_Form_MyExtension {\r\n    public void init() {\r\n      next init();\r\n      // Post-init logic\r\n    }\r\n  }\r\n\r\nForm data source CoC:\r\n  [ExtensionOf(formDataSourceStr(SalesTable, SalesTable))]\r\n  final class SalesTable_SalesTableDS_MyExtension {\r\n    public void init() {\r\n      next init();\r\n    }\r\n    public int active() {\r\n      int ret = next active();\r\n      return ret;\r\n    }\r\n  }\r\n\r\nForm data field CoC:\r\n  [ExtensionOf(formDataFieldStr(SalesTable, SalesTable, SalesId))]\r\n  final class SalesTable_SalesId_MyExtension {\r\n    public void modified() {\r\n      next modified();\r\n    }\r\n  }\r\n\r\nForm control CoC:\r\n  [ExtensionOf(formControlStr(SalesTable, ButtonSubmit))]\r\n  final class SalesTable_ButtonSubmit_MyExtension {\r\n    public void clicked() {\r\n      next clicked();\r\n    }\r\n  }\r\n\r\nData entity CoC:\r\n  [ExtensionOf(tableStr(CustCustomerEntity))]\r\n  final class CustCustomerEntity_MyExtension {\r\n    public boolean validateWrite() {\r\n      boolean ret = next validateWrite();\r\n      // Additional validation\r\n      return ret;\r\n    }\r\n  }\r\n\r\nStatic method wrapping:\r\n  [ExtensionOf(classStr(MyClass))]\r\n  final class MyClass_MyExtension {\r\n    public static MyClass construct() {\r\n      MyClass ret = next construct();\r\n      return ret;\r\n    }\r\n  }\r\n\r\nCoC RULES:\r\n  - Extension class MUST be final.\r\n  - MUST call next exactly once (unless method is [Replaceable]).\r\n  - Cannot call next conditionally (no if/while/for around the next call).\r\n  - CAN wrap the next call in try/catch/finally.\r\n  - Default parameter values in the wrapper MUST NOT include the default — pass them through.\r\n  - Cannot wrap constructors (new method).\r\n  - [Hookable(false)] prevents BOTH CoC wrapping AND event subscription.\r\n  - [Wrappable(false)] prevents CoC but still allows events.\r\n  - [Replaceable] allows skipping the next call (conditionally replacing the method).\r\n  - Access: can use this. to access public/protected members of the base object.\r\n  - In form CoC: use this. for element-level access (e.g. this.design(), this.args()).\r\n\r\n────────────────────────────────────────────────────────\r\n3. EXTENSION METHODS\r\n────────────────────────────────────────────────────────\r\nAdd new methods to existing types WITHOUT CoC. The class doesn't wrap anything —\r\nit adds net-new callable methods. PLUG-IN / FACTORY — Inheritance & Dependency Injection\r\n────────────────────────────────────────────────────────\r\nD365FO uses the Strategy pattern extensively. It has its own syntax rules:\r\n  - Table buffer variable is declared BEFORE the select, not in the select.\r\n  - No FROM keyword for single-table selects: select custTable where ...\r\n  - Use FROM only for the joined tables after the first data source.\r\n  - Field comparisons use == (not =), != (not <>), >= , <= , > , <\r\n  - Logical operators: && (AND), || (OR) — with parentheses for grouping.\r\n  - String literals use double quotes: \"value\"\r\n  - Enum values use EnumType::Value syntax: NoYes::Yes\r\n  - No semicolon after where clause parts — semicolon at the end of the full statement.\r\n\r\n  Single record:\r\n    select firstonly custTable\r\n        where custTable.AccountNum == \"US-001\";\r\n\r\n  Loop over records:\r\n    while select custTable\r\n        where custTable.CustGroup == \"10\"\r\n    {\r\n        info(custTable.AccountNum);\r\n    }\r\n\r\n  Multiple conditions:\r\n    select firstonly custTable\r\n        where custTable.CustGroup == \"10\"\r\n           && custTable.Blocked  == CustVendorBlocked::No;\r\n\r\n  Ordering:\r\n    while select custTable\r\n        order by custTable.AccountNum asc\r\n        where custTable.CustGroup == \"10\"\r\n    {\r\n        // ...\r\n    }\r\n\r\n  Aggregates:\r\n    select count(RecId) from custTable\r\n        where custTable.CustGroup == \"10\";\r\n    int64 cnt = custTable.RecId;  // aggregate result is in the field\r\n\r\n    select sum(CreditMax) from custTable;\r\n    real total = custTable.CreditMax;\r\n\r\n  Joins (2 tables):\r\n    while select custTable\r\n        join custTrans\r\n            where custTrans.AccountNum == custTable.AccountNum\r\n               && custTrans.TransDate  >= fromDate\r\n    {\r\n        // custTable and custTrans are both populated\r\n    }\r\n\r\n  Deep join chains (3+ tables — COMMON in D365FO due to heavy normalization):\r\n    D365FO data models routinely require 4–8 table joins. Each join level adds\r\n    another table buffer and its own where clause. It's in another of your own models (different package) → you CAN modify at source,\r\n     but consider extending from the consuming model if dependency direction matters.\r\n  e.
