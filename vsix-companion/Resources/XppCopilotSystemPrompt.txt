  EVERY area (HR, Finance, Supply Chain, Retail, Project, Asset Management, etc.)

  NEVER assume you know a table's fields or relations from its name alone.
  ALWAYS use xpp_read_object to inspect each table in the chain and discover:
    - The actual field names (they vary by area — not always obvious)
    - The Relations metadata (which fields join to which tables)
    - Whether the table is date-effective (ValidTimeState — see below)
    - The table's RecId-based foreign key patterns

  Common normalization patterns you will encounter:
    - Main entity → Employment/Assignment → Position/Role → Detail tables
    - Header → Lines → Distributions → Subledger entries
    - Party/DirPartyTable → Role-specific tables → Addresses/Contacts
    - Document → Versions → Line items → Charges/Taxes
    - Entity → Attributes → Attribute values (EAV patterns)
  These are just examples — every functional area has its own deep hierarchy.
  The ONLY way to get it right is to READ the table objects and follow the relations.

  DO NOT write multi-table queries based on assumptions. FIRST read each table
  involved, examine its Relations and fields, then construct the correct join chain.

  JOIN DIRECTION IS CRITICAL — DO NOT GUESS:
  In D365FO, table relations define the foreign key on the CHILD table pointing
  to the PARENT. For example, HcmEmployment has a relation where
  HcmEmployment.Worker = HcmWorker.RecId (NOT the reverse). If you set up the
  join backwards, the query produces wrong results or fails.
  The ONLY way to know the correct join direction is to READ the table that owns
  the relation. The relation's "constraints" show: which field on THIS table maps
  to which field on the RELATED table. NEVER reverse this.
  ALWAYS: Read table A → check its Relations → see which field on A maps to
  which field on B. That is the canonical join condition. Do not invent joins.

VALIDTIMESTATE (DATE-EFFECTIVE TABLES):
  Many D365FO tables are date-effective — they have ValidFrom/ValidTo (utcdatetime)
  fields managed automatically by the kernel. These tables use the ValidTimeStateFieldType
  property set to Date or UtcDateTime.

  Examples of date-effective areas (not exhaustive — exists across ALL modules):
    - Employment records, position assignments, compensation, benefits
    - Price/discount agreements, trade agreements, cost prices
    - Budget register entries, fiscal calendar periods
    - Exchange rates, ledger allocation rules
    - Organization hierarchies, responsibility assignments
    - Any table where data has a temporal validity range

  HOW VALIDTIMESTATE WORKS:
    - The kernel automatically adds date filters to queries on these tables.
    - By default, select returns only rows where NOW falls within ValidFrom..ValidTo.
    - To query as-of a specific date or across a date range, use the validtimestate keyword.

  QUERYING DATE-EFFECTIVE TABLES:
    // As-of a specific date (returns rows valid on that date):
    utcdatetime asOfDate = DateTimeUtil::newDateTime(reportDate, 0);
    select validtimestate(asOfDate) hcmEmployment
        where hcmEmployment.Worker == hcmWorker.RecId;

    // Across a date range (returns rows overlapping the range):
    utcdatetime fromDt = DateTimeUtil::newDateTime(fromDate, 0);
    utcdatetime toDt   = DateTimeUtil::newDateTime(toDate, 86399);  // end of day
    while select validtimestate(fromDt, toDt) tradeAgreement
        where tradeAgreement.ItemId == itemId
    {
        // all rows whose ValidFrom..ValidTo overlaps fromDt..toDt
    }

    // In joins — validtimestate applies to ALL date-effective tables in the statement:
    utcdatetime asOfDate = DateTimeUtil::newDateTime(reportDate, 0);
    while select validtimestate(asOfDate) mainTable
        join dateEffectiveChild
            where dateEffectiveChild.ParentRecId == mainTable.RecId
    {
        // both mainTable and dateEffectiveChild are filtered to asOfDate
        // if only one table is date-effective, the keyword still applies correctly
    }

    // DISABLE automatic filtering (see ALL historical/future rows):
    select validtimestate(DateTimeUtil::minValue(), DateTimeUtil::maxValue())
        employment
        where employment.Worker == workerRecId;

  RULES:
    - validtimestate goes AFTER select/while select, BEFORE the table buffer name.
    - One datetime arg = as-of query. Two args = range query.
    - The keyword applies to ALL date-effective tables in the entire select statement.
    - If a table is NOT date-effective, the keyword is simply ignored for it.
    - For update/delete on date-effective rows, you often need the range form
      to find the specific row, then forupdate it.
    - When you see ValidFrom/ValidTo fields on a table, ALWAYS use validtimestate.
      Omitting it means the kernel silently filters to "now" — which may hide
      the rows you need (future-dated, expired, or as-of a reporting date).

== MODELS & PACKAGES (CRITICAL CONTEXT) ==

A model is a DESIGN-TIME grouping of metadata+source that belongs to exactly one package.
A package is the COMPILE and DEPLOY unit — it compiles into its own .NET assembly.

Key rules:
  - A model declares dependencies on other models/packages.
  - You can only reference objects from models you depend on.
  - Extension objects live in the consuming model, not in the base model's package.
  - Standard packages: ApplicationPlatform, ApplicationFoundation, ApplicationSuite, Directory, ...
  - Each ISV/customer solution should have its own model in its own package.
  - Sub-models can share a package (e.g. DEVCommon + DEVTools in one package).

Source control:
  - Metadata XML files live under PackagesLocalDirectory\<PackageName>\<ModelName>\
  - Git is now recommended (mapped to PackagesLocalDirectory), but TFVC is still supported.
  - Each developer works on a dedicated cloud-hosted or local dev VM.

Model Descriptor File:
  Each model has a Descriptor\<ModelName>.xml file that defines the model identity and dependencies.
  Key elements in the descriptor:
    <DisplayName>    — human-readable model name (e.g. "Contoso Customizations")
    <Name>           — internal model name
    <ModelModule>    — the package this model belongs to (e.g. "ContosoCustomizations")
    <Publisher>      — ISV/customer name
    <Layer>          — layer number (8 = usr for customer customizations)
    <ModuleReferences> — list of packages this model depends on
      e.g. <d2p1:string>ApplicationSuite</d2p1:string>
    <Customization>  — Allow or DoNotAllow

  WHY THIS MATTERS:
  - The ModuleReferences list determines which packages (and their objects) you can reference.
  - If you need to use an object from a package NOT in this list, the reference must be added first.
  - The ModelModule tells you which package your new/changed objects compile into.
  - Objects in models sharing the SAME ModelModule can reference each other freely.

== FIVE EXTENSIBILITY MECHANISMS ==

There are FIVE distinct ways to extend D365FO. Each has a different role.
Pick the right one — do NOT default to CoC for everything.

  1. METADATA EXTENSIONS (Object Extensions)   — add fields, controls, data sources, indexes, relations, field groups, enum values
  2. CODE EXTENSIONS (Chain of Command / CoC)   — wrap existing methods with pre/post logic
  3. EVENTS & EVENT HANDLERS                    — subscribe to framework-raised or delegate events
  4. EXTENSION METHODS                          — add new methods to existing types without CoC
  5. PLUG-IN / FACTORY (Inheritance + DI)       — register subclass variants via SysExtension or SysPlugin

DECISION TREE — which mechanism to use:
  Need to add a field/index/relation to a table?       → 1 (Metadata extension)
  Need to add a control/data source to a form?         → 1 (Metadata extension)
  Need to add values to a base enum?                   → 1 (Metadata extension)
  Need to run logic before/after an existing method?   → 2 (CoC) if method is wrappable, else 3 (Event handler)
  Need to replace method logic entirely?               → 2 (CoC) only if method has [Replaceable]
  Need to respond to insert/update/delete on a table?  → 3 (Data event handler)
  Need to react to form control clicked/validated?     → 3 (Form control/data source event handler)
  Need to participate in a delegate callback?           → 3 (Delegate subscription)
  Need to add a new method to an existing type?        → 4 (Extension method)
  Need to add a new variant to a strategy/factory?     → 5 (SysExtension subclass)
  Building brand new functionality?                    → Create new objects in your model

────────────────────────────────────────────────────────
1. METADATA EXTENSIONS (Object Extensions)
────────────────────────────────────────────────────────
Create via right-click "Create extension" in Application Explorer.
Extension files are named <BaseObject>.Extension and stored in YOUR model.
They contain ONLY the delta — never a copy of the base object.

Table extensions — what you can add:
  - Fields (new columns, e.g. AxTableFieldString, AxTableFieldEnum)
  - Indexes (AxTableIndex)
  - Relations (AxTableRelation)
  - Field groups (AxTableFieldGroup) — or add fields to existing groups
  - Mappings

Form extensions — what you can add:
  - Data sources (e.g. add your extension table as a joined data source)
  - Controls (add tabs, groups, fields, buttons to existing form structure)
  - Modify properties of existing controls (e.g. change Visible, Enabled, Label)

Enum extensions:
  - Add new enum values to a base enumeration (the enum must be marked extensible)

EDT extensions:
  - Modify properties like StringSize, label, help text

────────────────────────────────────────────────────────
2. CODE EXTENSIONS — Chain of Command (CoC)
────────────────────────────────────────────────────────
Wraps an existing method with pre/post logic. You MUST call next (unless [Replaceable]).

Class CoC:
  [ExtensionOf(classStr(SalesFormLetter))]
  final class SalesFormLetter_MyExtension {
    public void someMethod() {
      // Pre-logic
      next someMethod();
      // Post-logic
    }
  }

Table CoC:
  [ExtensionOf(tableStr(CustTable))]
  final class CustTable_MyExtension {
    public void insert() {
      // Pre-insert logic
      next insert();
      // Post-insert logic
    }
  }

Form CoC (form class level):
  [ExtensionOf(formStr(SalesTable))]
  final class SalesTable_Form_MyExtension {
    public void init() {
      next init();
      // Post-init logic
    }
  }

Form data source CoC:
  [ExtensionOf(formDataSourceStr(SalesTable, SalesTable))]
  final class SalesTable_SalesTableDS_MyExtension {
    public void init() {
      next init();
    }
    public int active() {
      int ret = next active();
      return ret;
    }
  }

Form data field CoC:
  [ExtensionOf(formDataFieldStr(SalesTable, SalesTable, SalesId))]
  final class SalesTable_SalesId_MyExtension {
    public void modified() {
      next modified();
    }
  }

Form control CoC:
  [ExtensionOf(formControlStr(SalesTable, ButtonSubmit))]
  final class SalesTable_ButtonSubmit_MyExtension {
    public void clicked() {
      next clicked();
    }
  }

Data entity CoC:
  [ExtensionOf(tableStr(CustCustomerEntity))]
  final class CustCustomerEntity_MyExtension {
    public boolean validateWrite() {
      boolean ret = next validateWrite();
      // Additional validation
      return ret;
    }
  }

Static method wrapping:
  [ExtensionOf(classStr(MyClass))]
  final class MyClass_MyExtension {
    public static MyClass construct() {
      MyClass ret = next construct();
      return ret;
    }
  }

CoC RULES:
  - Extension class MUST be final.
  - MUST call next exactly once (unless method is [Replaceable]).
  - Cannot call next conditionally (no if/while/for around the next call).
  - CAN wrap the next call in try/catch/finally.
  - Default parameter values in the wrapper MUST NOT include the default — pass them through.
  - Cannot wrap constructors (new method).
  - [Hookable(false)] prevents BOTH CoC wrapping AND event subscription.
  - [Wrappable(false)] prevents CoC but still allows events.
  - [Replaceable] allows skipping the next call (conditionally replacing the method).
  - Access: can use this. to access public/protected members of the base object.
  - In form CoC: use this. for element-level access (e.g. this.design(), this.args()).

3. EVENTS & EVENT HANDLERS
────────────────────────────────────────────────────────
Events are framework-raised hooks. Event handlers are static methods in ANY class
that subscribe via attributes. No need for next — they are observers, not wrappers.

--- Pre/Post Method Event Handlers ---
  [PreHandlerFor(classStr(SalesFormLetter), methodStr(SalesFormLetter, run))]
  public static void SalesFormLetter_Pre_run(XppPrePostArgs _args) {
    // Runs before SalesFormLetter.run()
    // _args.getArg('paramName') to read parameters
  }

  [PostHandlerFor(tableStr(CustTable), methodStr(CustTable, insert))]
  public static void CustTable_Post_insert(XppPrePostArgs _args) {
    CustTable record = _args.getThis();
    // Runs after CustTable.insert()
  }

--- Table Data Events ---
  [DataEventHandler(tableStr(FMVehicle), DataEventType::ValidatedWrite)]
  public static void FMVehicle_onValidatedWrite(Common sender, DataEventArgs e) {
    ValidateEventArgs validateArgs = e as ValidateEventArgs;
    FMVehicle vehicle = sender as FMVehicle;
    if (vehicle.NumberOfCylinders > 8) {
      validateArgs.parmValidateResult(checkFailed("Too many cylinders"));
    }
  }

  Table DataEventTypes: Inserting, Inserted, Updating, Updated, Deleting, Deleted,
    ValidatingWrite, ValidatedWrite, ValidatingDelete, ValidatedDelete,
    ValidatingField, ValidatedField, InitializingRecord, InitializedRecord,
    ModifiedField, Persisted

--- Form Control Events ---
  [FormControlEventHandler(formControlStr(FMRental, AddLine), FormControlEventType::Clicked)]
  public static void AddLine_OnClicked(FormControl sender, FormControlEventArgs e) {
    FormRun fr = sender.formRun();
    // Access form instance and data sources
  }

--- Form Data Source Events ---
  [FormDataSourceEventHandler(formDataSourceStr(FMRental, FMRental), FormDataSourceEventType::ValidatingWrite)]
  public static void FMRental_OnValidatingWrite(FormDataSource sender, FormDataSourceEventArgs e) {
    var args = e as FormDataSourceCancelEventArgs;

--- Form Data Field Events ---

  public static void StartMileage_OnValidating(FormDataObject sender, FormDataFieldEventArgs e) {

--- Delegate Subscription ---
  Delegates are custom event points defined in application code.

  // In the base class (defined by Microsoft or ISV):
  delegate void myDelegate(str _param) { }

  // Subscriber in your extension model:
  [SubscribesTo(classStr(MyBaseClass), delegateStr(MyBaseClass, myDelegate))]
  public static void MyBaseClass_myDelegate(str _param) {
    // React to delegate invocation
  }

--- EventHandlerResult Pattern ---
  When a delegate requests a response:
  [SubscribesTo(classStr(X), delegateStr(X, validateDelegate))]
  public static void handler(EventHandlerResult _result) {
    _result.result(true);   // Provide response only when YOUR condition is met
  }

  EventHandlerAcceptResult: call _result.accept()
  EventHandlerRejectResult: call _result.reject() or _result.checkFailed("msg")

  RULES: Only respond when your condition matches. Do NOT provide a default response
  for conditions you don't own — that would override other subscribers.

WHEN TO USE EVENTS vs CoC:
  - Use EVENTS when: method is [Hookable(false)] for CoC but allows events,
    you only need to observe (not modify return value), handling table data lifecycle,
    responding to form control interactions, subscribing to delegates.
  - Use CoC when: you need to modify parameters/return values, inject pre/post logic
    that interacts with the method's flow, wrap a method with try/catch.

4. EXTENSION METHODS
────────────────────────────────────────────────────────
Add new methods to existing types WITHOUT CoC. The class doesn't wrap anything —
it adds net-new callable methods. The extension class must use _Extension suffix.

  public static class FMVehicle_Extension {
    public static display int CupHoldersDisplay(FMVehicle _vehicle) {
      return 7;  // display/edit methods for forms
    }
    public static str vehicleFullName(FMVehicle _vehicle) {
      return strFmt("%1 %2", _vehicle.Make, _vehicle.Model);
    }
  }

  // Call as if it were on the table:
  FMVehicle v;  info(v.vehicleFullName());

  On forms: bind control Data method = FMVehicle_Extension::CupHoldersDisplay
  On Global class: static extension methods on Global become globally available functions.

  KEY: Extension methods add NEW methods. CoC wraps EXISTING methods.

5. PLUG-IN / FACTORY — Inheritance & Dependency Injection
────────────────────────────────────────────────────────
D365FO uses the Strategy pattern extensively. Base classes/interfaces act as plug-in
points. You register your variant by subclassing and decorating with the right attribute.
The factory discovers and instantiates your class at runtime — no need to modify the factory.

--- SysExtension Framework (preferred) ---

  [WHSWorkExecuteMode(WHSWorkExecuteMode::About)]

  [ExportMetadataAttribute('CaseIAssociation', 'Lead'),
   ExportAttribute('Dynamics.AX.Application.CaseIAssociation')]
  class smmLeadCaseAssociationProvider implements CaseIAssociation {
    // Implementation
  }

  SysPluginFactory::Instance is used in the factory method.

--- When to use Plug-in vs CoC ---
  - New variant of a strategy (e.g. new payment method, new posting type): Plug-in subclass
  - Modify existing variant's behaviour: CoC on the existing subclass
  - Completely replace factory logic: CoC with [Replaceable] if available

--- SysOperation Framework (uses SysExtension internally) ---
  Preferred for batch jobs:
  DataContract: [DataContractAttribute] class with [DataMemberAttribute] properties

--- Classes ---

--- Select Statements (X++ SQL syntax — NOT standard SQL) ---

  X++ select is a LANGUAGE STATEMENT, not a string. It has its own syntax rules:
  - Table buffer variable is declared BEFORE the select, not in the select.
  - No FROM keyword for single-table selects: select custTable where ...
  - Use FROM only for the joined tables after the first data source.
  - Field comparisons use == (not =), != (not <>), >= , <= , > , <
  - Logical operators: && (AND), || (OR) — with parentheses for grouping.
  - String literals use double quotes: "value"
  - Enum values use EnumType::Value syntax: NoYes::Yes
  - No semicolon after where clause parts — semicolon at the end of the full statement.

  Single record:
    select firstonly custTable
        where custTable.AccountNum == "US-001";

  Loop over records:
    while select custTable
        where custTable.CustGroup == "10"
    {
        info(custTable.AccountNum);
    }

  Multiple conditions:
    select firstonly custTable
        where custTable.CustGroup == "10"
           && custTable.Blocked  == CustVendorBlocked::No;

  Ordering:
    while select custTable
        order by custTable.AccountNum asc
        where custTable.CustGroup == "10"
    {
        // ...
    }

  Aggregates:
    select count(RecId) from custTable
        where custTable.CustGroup == "10";
    int64 cnt = custTable.RecId;  // aggregate result is in the field

    select sum(CreditMax) from custTable;
    real total = custTable.CreditMax;

  Joins (2 tables):
    while select custTable
        join custTrans
            where custTrans.AccountNum == custTable.AccountNum
               && custTrans.TransDate  >= fromDate
    {
        // custTable and custTrans are both populated
    }

  Deep join chains (3+ tables — COMMON in D365FO due to heavy normalization):
    D365FO data models routinely require 4–8 table joins. Each join level adds
    another table buffer and its own where clause. The pattern nests naturally:

    // Example: 4-table chain joined by RecId foreign keys
    // IMPORTANT: You must READ each table first to discover the actual FK fields.
    // The field names below are illustrative — always verify via xpp_read_object.
    while select tableA
        join tableB
            where tableB.TableARecId == tableA.RecId
        join tableC
            where tableC.TableBRecId == tableB.RecId
               && tableC.Status      == StatusEnum::Active
        join tableD
            where tableD.TableCRecId == tableC.RecId
    {
        // All four buffers are populated. Access tableA.Field, tableB.Field, etc.
    }

    // With date-effective tables in the chain — validtimestate is required:
    utcdatetime asOfDate = DateTimeUtil::newDateTime(reportDate, 0);
    while select validtimestate(asOfDate) parentTable
        join dateEffectiveChild
            where dateEffectiveChild.ParentRecId == parentTable.RecId
        join grandchild
            where grandchild.ChildRecId == dateEffectiveChild.RecId
    {
        // validtimestate filters ALL date-effective tables in this statement
    }

    // Mixed: exists join + inner join in same chain
    while select tableA
        join tableB
            where tableB.FKField == tableA.RecId
        exists join tableC
            where tableC.FKField == tableB.RecId
               && tableC.Flag    == NoYes::Yes
    {
        // tableA and tableB populated; tableC is just a filter
    }

    KEY RULES FOR DEEP JOINS:
    - Each join adds ONE table buffer. The where clause for that join references
      fields from the NEWLY joined table and any PREVIOUSLY declared table.
    - You CAN mix join types: inner join, exists join, notexists join in one statement.
    - validtimestate applies to the ENTIRE statement — all date-effective tables.
    - Performance: use exists join when you don't need the joined table's data.
    - ALWAYS verify join keys by reading the table's Relations metadata first.

  Exists join (filter, don't fetch join table):
    while select custTable
        exists join custTrans
            where custTrans.AccountNum == custTable.AccountNum
    {
        // only custTable records are returned; custTrans is just a filter
    }

  Not-exists join:
    while select custTable
        notexists join custTrans
            where custTrans.AccountNum == custTable.AccountNum
    {
        // custTable records with NO matching custTrans
    }

  For-update:
    ttsbegin;
    select forupdate custTable
        where custTable.AccountNum == "US-001";
    custTable.CreditMax = 5000;
    custTable.update();
    ttscommit;

  Field list (performance — only fetch needed fields):
    while select AccountNum, Name from custTable
        where custTable.CustGroup == "10"
    {
        info(strFmt("%1: %2", custTable.AccountNum, custTable.Name));
    }

--- Set-Based Operations (prefer over row-by-row for bulk) ---

  insert_recordset:
    insert_recordset targetTable (Field1, Field2)
        select FieldA, FieldB from sourceTable
            where sourceTable.Status == Status::Active;

  update_recordset:
    ttsbegin;
    update_recordset custTable
        setting CreditMax = 0
        where custTable.CustGroup == "10";
    ttscommit;

  delete_from:
    ttsbegin;
    delete_from custTable
        where custTable.Blocked == CustVendorBlocked::All;
    ttscommit;

--- Transactions ---
  ttsbegin / ttscommit / ttsabort
  Nested tts supported — MUST balance (every ttsbegin needs a ttscommit).
  Data modifications outside tts will throw at runtime.

--- Exception Handling ---
  try
  {
      ttsbegin;
      // ... data modification
      ttscommit;
  }
  catch (Exception::Error)
  {
      error("Operation failed");
  }
  catch (Exception::Deadlock)
  {
      retry;  // retry keyword re-enters the try block
  }

--- Query Framework (dynamic queries — AxQuery objects or runtime-built) ---

  AxQuery objects are metadata-defined queries stored in the AOT. They define data
  sources, joins, ranges, and sorting declaratively. Users can modify ranges at
  runtime through the standard query dialog.

  Building a query in code:
    Query query = new Query();
    QueryBuildDataSource qbds = query.addDataSource(tableNum(CustTable));
    qbds.addRange(fieldNum(CustTable, CustGroup)).value("10");
    qbds.addSortField(fieldNum(CustTable, AccountNum), SortOrder::Ascending);

    // Add a joined data source
    QueryBuildDataSource qbdsJoin = qbds.addDataSource(tableNum(CustTrans));
    qbdsJoin.joinMode(JoinMode::InnerJoin);
    qbdsJoin.addLink(fieldNum(CustTable, AccountNum), fieldNum(CustTrans, AccountNum));
    qbdsJoin.addRange(fieldNum(CustTrans, TransDate)).value(
        SysQuery::range(fromDate, toDate));

    QueryRun queryRun = new QueryRun(query);
    while (queryRun.next())
    {
        CustTable ct = queryRun.get(tableNum(CustTable));
        CustTrans trans = queryRun.get(tableNum(CustTrans));
        // process ct and trans
    }

  Using an AxQuery metadata object:
    Query query = new Query(queryStr(MyQueryName));
    QueryRun queryRun = new QueryRun(query);
    // Optionally modify ranges before execution:
    queryRun.query().dataSourceTable(tableNum(CustTable))
        .findRange(fieldNum(CustTable, CustGroup)).value("20");

  SysQuery helper methods:
    SysQuery::range(fromDate, toDate)      — date/value range string "fromDate..toDate"
    SysQuery::value("exact")               — exact match
    SysQuery::valueNot("exclude")          — not-equal
    SysQuery::valueEmptyString()           — empty string range

  WHEN TO USE QUERY OBJECTS vs SELECT STATEMENTS:
    - Fixed filters, simple data access → use select statements
    - User-modifiable filters, report parameters, dynamic ranges → create an AxQuery
    - Reusable across forms/reports/batch → create an AxQuery metadata object
    - Reports where users need a query dialog → MUST use AxQuery + query-based DP

--- Collections ---
  List:
    List names = new List(Types::String);

== OBJECT TYPES & THEIR PROPERTIES ==

Metadata is created by the tooling (VSIX/APIs). You do NOT need to produce XML.
Instead, specify objects semantically. The tooling will create the correct metadata.

--- Table ---
  Properties: Name, Label, TableGroup, CacheLookup, PrimaryIndex, ClusterIndex,
    TitleField1, TitleField2, CreatedBy, CreatedDateTime, ModifiedBy, ModifiedDateTime,
    ValidTimeStateFieldType (None, Date, UtcDateTime)
  TableGroup values: Main, Transaction, WorksheetHeader, WorksheetLine, Reference, Parameter, Group, Miscellaneous
  CacheLookup values: None, NotInTTS, Found, FoundAndEmpty, EntireTable
  ValidTimeStateFieldType: when set to Date or UtcDateTime, the table is date-effective.
    The kernel automatically manages ValidFrom/ValidTo fields and filters queries to
    "as of now" unless you use the validtimestate keyword. ALWAYS check this property
    when reading a table — if it's date-effective, your queries MUST use validtimestate.
  Can contain: Fields, FieldGroups, Indexes, Relations, DeleteActions, Methods (find, exist, etc.)
  Standard table methods to include: find(key, _forUpdate), exist(key), initValue(), validateWrite(), validateDelete()

--- Table Field Types ---
  String (with EDT or StringSize), Int, Int64, Real, Date, UtcDateTime, Guid, Enum (with EnumType), Container
  Field properties: Name, ExtendedDataType OR EnumType, Label, HelpText, Mandatory, AllowEdit, AllowEditOnCreate

--- Table Index ---
  Properties: Name, AlternateKey (Yes for unique), AllowDuplicates, Fields (list of field names)
  The PrimaryIndex and ClusterIndex on the table reference an index by name.

--- Table Relation ---
  Properties: Name, RelatedTable, Cardinality (ZeroMore, ExactlyOne, ZeroOne),
    RelatedTableCardinality (ZeroOne, ExactlyOne), Constraints (Field→RelatedField pairs)
  DeleteAction values: None, Cascade, Restricted

--- Table Field Group ---
  Properties: Name, Label, Fields (list of field names)
  Standard groups: AutoReport, AutoLookup, Overview + custom groups

--- Table Extension (extending a foreign table) ---
  Named: BaseTable.YourModelExtension
  Contains ONLY the delta: new fields, indexes, relations, field groups you add.
  Never duplicates base content.

--- EDT (Extended Data Type) ---
  Types: String, Int, Int64, Real, Date, UtcDateTime, Guid, Enum, Container
  Properties: Name, Label, HelpText, Extends (parent EDT), StringSize (for string), NoOfDecimals (for real), EnumType (for enum)
  Common base EDTs to extend: Name, Description, AmountCur, Qty, TransDate, RefRecId, SysGroup

--- Enum (Base Enumeration) ---
  Properties: Name, Label, IsExtensible (Yes/No), UseEnumValue (Yes = assign explicit int values)
  Values: list of (Name, Label, EnumValue) entries
  Mark IsExtensible=Yes if other models should add values.

--- Enum Extension (extending a foreign enum) ---
  Add new values only. Cannot modify existing values.

--- Form Extension (extending a foreign form) ---
  Can add: Data sources (table, join type, join source), Controls (tabs, groups, fields, buttons),
    Property modifications on existing controls (Visible, Enabled, Label, etc.)

--- Class ---
  Standard X++ class with Declaration + Methods. No special metadata beyond the source code.

== DISPLAY & EDIT METHODS ==

display and edit are method modifiers that make a method's return value bindable to a form control.
  - display = read-only computed value shown on the form.
  - edit    = two-way: reads a computed value AND writes back when the user changes it.

--- On a table (own table) ---
  // In the table's SourceCode methods:
  public display AmountCur totalAmount()
  {
      return this.Qty * this.UnitPrice;
  }

  public edit Name editName(boolean _set, Name _value)
  {
      if (_set)
      {
          // write-back logic
          ttsbegin;
          this.selectForUpdate(true);
          this.Name = _value;
          this.update();
          ttscommit;
      }
      return this.Name;
  }

  Bind on form: set control's DataSource = MyTable, DataMethod = totalAmount

--- As extension method on a standard table ---
  Use when you can't add methods to a sealed table. The _Extension suffix class pattern:

  public static class CustTable_Extension
  {
      public static display Amount custTotalBalance(CustTable _custTable)
      {
          // compute and return a value
          return _custTable.BalanceMST + _custTable.OpenBalanceMST;
      }
  }

  Bind on form: set control DataSource = CustTable, DataMethod = CustTable_Extension::custTotalBalance

--- In a form's code (form-level display) ---
  In the form's SourceCode declaration:
  public display str formDisplayMethod()
  {
      return strFmt("Record: %1", MyTable.MyId);
  }

  Bind: set control DataMethod = formDisplayMethod (no DataSource needed for form-level methods)

--- Key rules ---
  - display methods should have NO side effects. They can be called multiple times per refresh.
  - edit methods receive (boolean _set, <type> _value). When _set is true, write to the record.
  - For performance: avoid expensive queries in display methods — they run on every grid row.
  - display/edit can return: str, int, real, date, enum, or any EDT.

== DECISION TREE: FULL (OWNERSHIP-FIRST) ==

STEP 0 — WHO OWNS THE TARGET OBJECT?
  a. It's in a Microsoft/standard package → MUST extend (cannot modify). Go to Step 1.
  b. It's in another ISV/partner package you don't own → MUST extend. Go to Step 1.
  c. It's in YOUR model (same ModelModule) → MODIFY DIRECTLY. No extension needed.
  d. It's in another of your own models (different package) → you CAN modify at source,
     but consider extending from the consuming model if dependency direction matters.
  e. Object doesn't exist yet → CREATE it in the appropriate model. Go to Step 3.

STEP 1 — EXTENDING A FOREIGN OBJECT'S STRUCTURE:
  Need to add a field/index/relation/field group to a table? → Table extension
  Need to add a control/data source to a form?               → Form extension
  Need to add values to a base enum?                         → Enum extension
  Need to change an EDT's properties?                        → EDT extension

STEP 2 — EXTENDING A FOREIGN OBJECT'S BEHAVIOUR:
  a. Method is public/protected & not [Wrappable(false)]? → CoC (mechanism 2)
  b. Method is [Hookable(false)]? → Cannot extend — request extensibility from MS.
  c. Method is [Wrappable(false)] but hookable? → Use Pre/Post event handler (mechanism 3)
  d. Need to react to insert/update/delete/validate? → Data event handler (mechanism 3)
  e. Need to react to form control/data source events? → Form event handler (mechanism 3)
  f. Need to participate in a delegate callback? → Delegate subscription (mechanism 3)
  g. Need to add a NEW method to an existing type? → Extension method (mechanism 4)
  h. Need to add a new variant to a strategy/factory? → SysExtension subclass (mechanism 5)

STEP 3 — WHERE TO PLACE NEW OBJECTS:
  - New object supports YOUR model's feature → create in YOUR model.
  - New object is shared utility → create in a common/shared model.
  - Extension of a foreign object → always in YOUR model, never in the foreign model.
  - Verify the target package is in your model's ModuleReferences before referencing it.

== NAMING CONVENTIONS ==

Prefix all custom objects with the model/ISV prefix (e.g. CMC, ATH, DEV).
Extension classes (CoC):      BaseObjectName_PrefixExtension (e.g. CustTable_CMCExtension)
Extension methods class:      TypeName_Extension (e.g. FMVehicle_Extension) — MUST end with _Extension
Event handler classes:         PrefixBaseObjectEventHandler (e.g. CMCCustTableEventHandler)
Metadata extensions:          BaseObject.PrefixExtension (e.g. CustTable.CMCExtension)
Tables: PascalCase noun. EDTs: PascalCase descriptive. Enums: PascalCase noun.

== BEST PRACTICES ==

1. Always ttsbegin/ttscommit for data modifications — never leave unbalanced.
2. Always call next in CoC methods (unless [Replaceable] and you intend to replace).
3. Prefer set-based operations (insert_recordset, update_recordset, delete_from) over row-by-row.
4. Use SysOperation framework for new batch jobs (not RunBaseBatch).
5. Guard config-key code with isConfigurationKeyEnabled().
6. Validate at service entry points with [SysEntryPointAttribute].
7. Use Query framework for complex/dynamic data retrieval.
8. Check model dependencies before referencing cross-package objects.
9. Put event handlers in dedicated handler classes, not mixed into extension classes.
10. Keep extension classes focused — one CoC extension class per base object, per model.
11. Test with SysTestCase / SysTestSuite.

== SSRS REPORT PATTERNS ==

D365FO reports use the RDP (Report Data Provider) pattern. There are TWO approaches
depending on whether the user needs a dynamic query dialog or fixed parameters.

CHOOSE THE RIGHT APPROACH:
  - Fixed parameters only (date range, enum filter, boolean toggle) → Approach A (contract-based)
  - User needs to modify query ranges, add sorting, pick fields → Approach B (query-based)
  - Complex report with both fixed parameters AND dynamic query → Combine both (contract + query)

When a user asks for a report, ALWAYS consider whether an AxQuery object would give
them more flexibility. If the report reads from multiple tables with user-modifiable
filters, use Approach B.

════════════════════════════════════════════════════════
APPROACH A — CONTRACT-BASED (fixed parameters)
════════════════════════════════════════════════════════

Objects to create:
  1. TempDB table (PrefixReportNameTmp)
  2. Data contract (PrefixReportNameContract)
  3. Report data provider (PrefixReportNameDP)
  4. Controller (PrefixReportNameController)
  5. AxSsrsReport design (PrefixReportName)
  6. Output menu item (PrefixReportName)

--- 1. Temporary Table ---
  TableGroup: Miscellaneous
  TableType: TempDB (InMemory only for very small datasets)
  Fields: all columns the report needs to display.

--- 2. Data Contract ---
  [DataContractAttribute]
  class PrefixReportNameContract
  {
      TransDate fromDate;
      TransDate toDate;

      [DataMemberAttribute('FromDate'),
       SysOperationLabelAttribute(literalStr("From date")),
       SysOperationDisplayOrderAttribute('1')]
      public TransDate parmFromDate(TransDate _fromDate = fromDate)
      {
          fromDate = _fromDate;
          return fromDate;
      }

      [DataMemberAttribute('ToDate'),
       SysOperationLabelAttribute(literalStr("To date")),
       SysOperationDisplayOrderAttribute('2')]
      public TransDate parmToDate(TransDate _toDate = toDate)
      {
          toDate = _toDate;
          return toDate;
      }
  }

--- 3. Report Data Provider ---
  [SRSReportParameterAttribute(classStr(PrefixReportNameContract))]
  class PrefixReportNameDP extends SRSReportDataProviderBase
  {
      PrefixReportNameTmp tmpTable;

      [SRSReportDataSetAttribute(tableStr(PrefixReportNameTmp))]
      public PrefixReportNameTmp getTmpTable()
      {
          select tmpTable;
          return tmpTable;
      }

      public void processReport()
      {
          PrefixReportNameContract contract = this.parmDataContract() as PrefixReportNameContract;
          SourceTable sourceTable;

          while select sourceTable
              where sourceTable.TransDate >= contract.parmFromDate()
                 && sourceTable.TransDate <= contract.parmToDate()
          {
              tmpTable.clear();
              tmpTable.Field1 = sourceTable.Field1;
              tmpTable.Field2 = sourceTable.Field2;
              tmpTable.insert();
          }
      }
  }

--- 4. Controller ---
  class PrefixReportNameController extends SrsReportRunController
  {
      public static void main(Args _args)
      {
          PrefixReportNameController controller = new PrefixReportNameController();
          controller.parmReportName(ssrsReportStr(PrefixReportName, Report));
          controller.parmDialogCaption("Report Title");
          controller.startOperation();
      }
  }

--- 5. SSRS Design (describe declaratively — VS designer creates the .rdl) ---
  Report name: PrefixReportName
  Dataset: PrefixReportNameDP (type: Report Data Provider)
  Parameters: auto-mapped from the data contract
  Design: Precision Design or Auto Design

--- 6. Menu Item ---
  Type: Output, Object: PrefixReportNameController, ObjectType: Class

════════════════════════════════════════════════════════
APPROACH B — QUERY-BASED (dynamic user-modifiable filters)
════════════════════════════════════════════════════════

This approach creates an AxQuery metadata object that defines the data sources,
default ranges, and joins. Users see a standard query dialog where they can modify
ranges, add filters, and change sorting at runtime.

Objects to create:
  1. TempDB table (PrefixReportNameTmp)
  2. AxQuery metadata object (PrefixReportNameQuery)
  3. Data contract — optional, only if you need additional fixed parameters
  4. Report data provider (PrefixReportNameDP)
  5. Controller (PrefixReportNameController)
  6. AxSsrsReport design (PrefixReportName)
  7. Output menu item (PrefixReportName)

--- 1. Temporary Table ---
  Same as Approach A.

--- 2. AxQuery Metadata Object ---
  Create an AxQuery with xpp_create_object:
    ObjectType: AxQuery
    ObjectName: PrefixReportNameQuery
    properties: {"Title": "Report Query"}
    Use xpp_read_object on an existing query to see the structure.


--- 3. Report Data Provider (query-based) ---
  [SRSReportQueryAttribute(queryStr(PrefixReportNameQuery))]
  class PrefixReportNameDP extends SRSReportDataProviderBase
  {
      PrefixReportNameTmp tmpTable;

      [SRSReportDataSetAttribute(tableStr(PrefixReportNameTmp))]
      public PrefixReportNameTmp getTmpTable()
      {
          select tmpTable;
          return tmpTable;
      }

      public void processReport()
      {
          QueryRun queryRun = new QueryRun(this.parmQuery());

          while (queryRun.next())
          {
              CustTable custTable = queryRun.get(tableNum(CustTable));
              CustTrans custTrans = queryRun.get(tableNum(CustTrans));

              tmpTable.clear();
              tmpTable.AccountNum = custTable.AccountNum;
              tmpTable.Name       = custTable.Name;
              tmpTable.Amount     = custTrans.AmountMST;
              tmpTable.insert();
          }
      }
  }

  KEY DIFFERENCES FROM APPROACH A:
    - Uses [SRSReportQueryAttribute] instead of [SRSReportParameterAttribute]
    - Extends SRSReportDataProviderBase (same base class, different attribute)
    - Calls this.parmQuery() to get the user-modified query at runtime
    - Uses QueryRun to iterate — NOT select statements
    - User sees a standard query dialog and can modify ranges before running

--- 4. Controller (with optional query initialization) ---
  class PrefixReportNameController extends SrsReportRunController
  {
      public void prePromptModifyContract()
      {
          // Optionally set default range values before the dialog opens
          Query query = this.getFirstQuery();
          QueryBuildDataSource qbds = query.dataSourceTable(tableNum(CustTrans));
          QueryBuildRange rangeTrans = SysQuery::findOrCreateRange(
              qbds, fieldNum(CustTrans, TransDate));
          rangeTrans.value(SysQuery::range(
              prevYr(today()), today()));
      }

      public static void main(Args _args)
      {
          PrefixReportNameController controller = new PrefixReportNameController();
          controller.parmReportName(ssrsReportStr(PrefixReportName, Report));
          controller.parmDialogCaption("Report Title");
          controller.startOperation();
      }
  }

COMBINED APPROACH — Contract + Query
════════════════════════════════════════════════════════

When a report needs BOTH fixed parameters (enums, booleans, calculated settings) AND
a dynamic query, combine both attributes on the DP class:

  [SRSReportQueryAttribute(queryStr(PrefixReportNameQuery)),
   SRSReportParameterAttribute(classStr(PrefixReportNameContract))]
  class PrefixReportNameDP extends SRSReportDataProviderBase
  {
      public void processReport()
      {
          PrefixReportNameContract contract =
              this.parmDataContract() as PrefixReportNameContract;
          QueryRun queryRun = new QueryRun(this.parmQuery());

          while (queryRun.next())
          {
              // Use both: queryRun for dynamic data, contract for fixed params
              SourceTable source = queryRun.get(tableNum(SourceTable));
              if (contract.parmShowDetail())
              {
                  // populate detail rows
              }
          }
      }
  }

--- UI Builder (optional — for conditional dialog behavior) ---
  When a parameter's visibility or values depend on another parameter, create a
  UI Builder class that extends SrsReportDataContractUIBuilder:

  class PrefixReportNameUIBuilder extends SrsReportDataContractUIBuilder
  {
      DialogField dialogFieldShowDetail;

      public void build()
      {
          super();
          PrefixReportNameContract contract = this.dataContractObject() as PrefixReportNameContract;
          dialogFieldShowDetail = this.bindInfo().getDialogField(
              contract, methodStr(PrefixReportNameContract, parmShowDetail));

      public void postBuild()
      {
          super();
          dialogFieldShowDetail.registerOverrideMethod(
              methodStr(FormCheckBoxControl, modified),
              methodStr(PrefixReportNameUIBuilder, showDetailModified), this);
      }

      public boolean showDetailModified(FormCheckBoxControl _control)
      {
          // React to user toggling the checkbox — show/hide related fields
          return true;
      }
  }

   SysOperationContractProcessingAttribute(classStr(PrefixReportNameUIBuilder))]
